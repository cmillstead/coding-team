# Codex Learnings — Pre-flight Checklist

Anti-patterns that Codex cross-model review has repeatedly caught in Claude's plans and diffs.
The `/second-opinion` pre-flight checks every entry here BEFORE dispatching, so Codex spends its
rounds on novel problems instead of the same recurring mistakes. The post-review learning-capture
loop appends new recurring patterns here with the next ID.

**How to use (pre-flight):** check the plan/diff against every `P##` and `C##` below. Classify each
into `✓` (checked, clean), `FIXED` (violation found and fixed before dispatch), or `N/A(reason)`.
See `SKILL.md` → "Pre-flight — MANDATORY" for the coverage-line format.

**Seeded 2026-06-07** from accumulated memory (`feedback_codex_second_opinion`,
`feedback_path-trust-tiers`, `feedback_codex-cross-model-value`). This is v0 — small and honest.
It grows via learning capture, not by inventing slots.

## P — Plan patterns

Plans written from memory of the code are optimistic. These are the mismatches Codex catches when
it reads the actual code. Round 1 is mostly structural; round 2 mostly precision (see SKILL.md
round budgeting).

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| P1 | **Phantom symbol** — plan references a field, type, function, or column that doesn't exist | Grep the codebase for every named symbol the plan touches. If it isn't there, the plan is describing code from memory, not reality. |
| P2 | **Wrong API name/shape** — wrong method name, signature, return type, or AST node name | Open the actual definition for every API the plan calls. Codesight/tree-sitter node names and SDK signatures are the usual offenders — confirm, don't recall. |
| P3 | **Stale command** — a shell/CLI command whose flags or subcommands no longer exist | Run `--help` (or check the CLI source) for every command the plan tells someone to run. Flags drift; verify the exact invocation. |
| P4 | **Wrong identifier** — nonexistent parameter, wrong variable name, or mis-scoped reference | Trace each variable/param named in the plan to its declaration. A name that "sounds right" but isn't there is a round-1 finding. |

| P30 | **A plan that REUSES/extends an existing shared helper assumes it WORKS as its name/signature implies, but the helper's BODY is a no-op or contract-violating — an `acquire()` that always returns `true`, a `lockName`/`scope` param accepted-but-ignored, a `validate()` that never rejects, a backing table whose `CHECK`/`UNIQUE`/`PK` forbids the new use.** The symbol EXISTS (P1 passes) and the signature FITS (P2 passes), so name+shape checks wave it through — but building on it inherits a silent defect. Caught across 2 rounds on the engram ADR-001 sync-lock plan: the plan said "reuse `SqliteLockProvider`, key on `lockName`, acquire 'sync'", but `SqliteLockProvider.acquire(lockName, ttlMs)` IGNORES `lockName` and hardcodes `INSERT OR IGNORE INTO consolidation_lock (id) VALUES (1)` then returns `true` UNCONDITIONALLY (never `false` on contention — broken as a mutex), with an existing test pinning that broken behavior; AND the target `consolidation_lock` had `id INTEGER PRIMARY KEY CHECK (id = 1)` in a DEPLOYED (immutable) migration, so per-name keying was structurally impossible without a new migration. The "reuse" would have shipped a lock that never locks. | For any plan that REUSES or EXTENDS an existing helper/utility/provider/table, open its BODY (not just its signature): (1) is every parameter actually USED inside the fn? (grep the param name in the body — an accepted-but-ignored arg is the tell); (2) does the return value VARY with outcome? (a mutex `acquire` must be able to return false; a `find` that always returns the first row; a guard that always passes); (3) does the backing DDL PERMIT the new use? — open the `CREATE TABLE`/constraints; a `CHECK`/`UNIQUE`/`PK (id=1)` may forbid it, and if the DDL lives in a DEPLOYED migration it is immutable (needs a NEW migration, never an edit); (4) does an existing TEST pin the current (possibly broken) behavior? — reuse means updating that test. "It already exists with the right signature" is P1/P2-clean but is NOT proof the body honors the contract its name advertises. Prefer a self-contained implementation over reusing a helper whose body you have not read. Pairs with P1 (phantom symbol — there the symbol is ABSENT; here it's PRESENT but lying) and C9 (type-valid-but-degenerate value). |
| P31 | **A metric/aggregate uses a `None`/skip sentinel for "the unit produced nothing," which silently EXCLUDES legitimate zero-score MISSES from the average — inflating it — because the same sentinel also means "not applicable." Distinguish "not applicable" from "zero-score miss" with an EXPLICIT eligibility/capability flag, not a value inferred from the empty output.** When a per-unit scorer returns `Option<Score>` and the aggregate macro-averages only the `Some` values, every `None` vanishes from the denominator. That is correct for units genuinely out-of-scope (a competitor that emits no symbols → N/A), but WRONG for an in-scope unit that simply scored zero (an eligible retriever that retrieved nothing → a real recall miss). Inferring the distinction from the empty output itself (`count==0`, empty list) conflates the two: a competitor and a failed/empty eligible run look identical. Caught on the Axon symbol-precision metric: all-anchor runs, ERRORED runs (error-fallback `symbol: None`), and zero-chunk successful runs were each dropped as `N/A` instead of counted as recall=0, inflating reported recall on exactly the hardest tasks. The fix was an explicit `emits_symbols` capability flag, so an eligible unit's empty output scores as a zero-miss while a non-eligible unit stays N/A. | For any `Option`-returning per-unit metric whose aggregate skips `None`, enumerate WHY a unit can be `None` and split the reasons into "out-of-scope / not-applicable" (correctly excluded) vs "in-scope but scored zero / failed / empty" (must be COUNTED as a zero). If the code infers this from the empty output (`count==0`, empty collection, a null dependency), it conflates them — add an EXPLICIT eligibility flag carried from where capability is known (the producer/trait), and decide zero-vs-N/A on that flag. Check EVERY path that produces the empty output — success-with-nothing, error-fallback, and any "no data" short-circuit — they are siblings of the same inflation. |
| P32 | **A GROUND/plan asserts a resource or convention does NOT exist (and builds the plan around creating it FRESH / inventing a new location) based on a single empty `find`/`grep` — but it DOES exist elsewhere with an established convention to mirror.** Negative existence claims get waved through with less rigor than positive ones (P1), yet they are just as load-bearing: "there are no fixtures, so we'll make a new dir" silently invents a parallel, inconsistent location. The single search that returned empty missed a sibling directory / different path root (a `-prune` that over-excluded, the wrong subtree). Caught on the Baxter shell-classifier plan: GROUND.md asserted "No `tests/fixtures/` anywhere" (from one pruned `find`), so the plan specified `src-tauri/tests/fixtures/…` + `include_str!(concat!(env!("CARGO_MANIFEST_DIR"), …))`; in fact repo-root `tests/fixtures/fnv-vectors.json` existed, loaded by `lib.rs` via `include_str!("../../tests/fixtures/…")` — the plan should have MIRRORED that proven convention, and `CARGO_MANIFEST_DIR` would have resolved to the wrong root. | For any "X does not exist / there is no established pattern for Y, so we'll create it fresh" claim, verify the ABSENCE with the same rigor as a presence (P1): don't trust one `find`/`grep` (wrong root, over-broad `-prune`, a different dir). Confirm by asking the CONSUMER — "how does a SIBLING test/module already load this kind of resource?" Grep for the capability (`include_str!`, fixture loaders, the file extension) across the WHOLE repo, not one subtree. If a convention exists, the plan MIRRORS it (same location, same load form) instead of inventing a parallel one. A false negative-existence claim makes the plan build duplicate/inconsistent infrastructure — the inverse of P1's phantom-symbol. |
| P33 | **A security/validation guard that resolves AMBIGUITY by enumerating candidate interpretations and DENYING if ANY candidate is dangerous over-denies inputs whose BENIGN interpretation was equally valid.** "A false-positive deny is free because the dangerous reading is malicious" is FALSE whenever a benign reading produces the same token shape — the guard then hard-blocks legitimate commands, eroding trust in the gate (the exact "the gate is dumb, stop trusting it" failure the gate exists to prevent). Caught on the Baxter exec-wrapper plan: a ±1 candidate model for unmodeled wrapper flags emitted two candidate command-positions and denied if either was a deny-family command; `timeout --weird echo sudo` (benign — `echo` prints the word "sudo") denied because `sudo` landed in a candidate slot, even though the real command was `echo`. The premise "the string is malicious" held for `timeout --weird 5 sudo reboot` but not for the token-identical benign case. | For any guard that DISAMBIGUATES by guessing among candidate parses/positions/interpretations, do NOT "deny if any candidate is dangerous" — that over-denies every input whose benign parse shares the shape. Deny ONLY on POSITIVE detection: an UNAMBIGUOUS resolution to a dangerous value. When the parse is ambiguous, fall to the safe NON-deny state — for a 3-way classifier that is ASK (human-gated), never DENY-on-a-guess and never silent ALLOW. Test the dominance property: the new rule must introduce ZERO over-deny relative to the prior behavior while only tightening positively-detected cases. The tell is a rationalization like "a false-positive deny is free / the string is malicious anyway" — verify a BENIGN input can't produce the same candidate shape before accepting it. |

## C — Code / diff patterns

Mistakes Codex catches in actual diffs that same-model audit rounds miss, because Claude's audit
reads the code with the assumptions that wrote it.

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C1 | **Single-gate path trust boundary** — validating a path-shaped field with one rule (e.g. `if value.contains("/")`) lets slashless inputs (`".."`, `"tmp"`, `"src"`) bypass the check and resolve against cwd. Caught as a CRITICAL slashless-bypass in the codesight MCP build. | Classify every path-shaped field into three tiers and validate accordingly: **(1) Identifier** (`repo` accepting `"engram"` OR an absolute path) — only path-form gets the prefix check. **(2) Filesystem path** (`path`, `repoPath`, `storagePath`) — ALWAYS `path.resolve` + require result under the trusted prefix, slash or not. **(3) Repo-relative** (`filePath`, `pathPrefix`) — reject any `..` segment via `path.posix.normalize` + segment split. Ask "identifier or filesystem path?" for each field; the rules differ. |
| C2 | **New `validate()`/guard rejects a value that is a valid sentinel elsewhere** — adding a "reject 0/empty/-1" rule without checking how the field is consumed. E.g. rejecting `rebuild_interval == 0` when `sync.rs` guards `if rebuild_interval > 0 { rebuild }`, so `0` means "disabled" — the validation turns a supported config into a startup error. Caught on the axon code-scan remediation diff (CODE-MED-5). | Before adding a "reject N" rule, grep every consumer of the field (`> 0`, `== 0`, `is_empty()`, `.unwrap_or`, match arms). If ANY consumer treats the rejected value as a sentinel/disable/default mode, do not reject it. A scan finding that says "reject 0" may be wrong about the value being invalid — verify against usage. |
| C3 | **CI step runs a venv-dependent tool without an active virtualenv** — `maturin develop`, `pip install -e .`, etc. fail on a clean runner because `actions/setup-python` provides an interpreter, NOT a venv. The job goes red on every push before tests run. Caught on the axon `python-bindings` CI job (TEST-HIGH-3). | Any new/edited CI step invoking `maturin develop` or other venv-requiring tools must create + activate a venv first (`python -m venv .venv && . .venv/bin/activate` in the SAME `run:` block, since each step is a fresh shell — or persist via `$GITHUB_PATH`/`$GITHUB_ENV`). You usually can't run the job locally, so this is a read-time check. |
| C4 | **A "lossy transform made safe by a full-fidelity stash/backup" feature breaks its own retrievability guarantee in three coupled ways — stashing an already-LOSSY projection instead of the raw source, appending the retrieval POINTER to a body that is then HEAD-truncated (clipping the pointer when output is largest), and running the destructive transform even when the stash write SILENTLY FAILED.** When a feature compresses/redacts/summarizes data and claims "the full version is retrievable via the stash," all three invariants must hold or the claim is false exactly in the worst case. Caught on the Baxter MCP-output `compact_json` plan: (1) it stashed an already-lossy text projection (non-text content → `"[type content omitted]"`, `structuredContent` dropped) instead of `serde_json::to_string(&res)`, so many results were unrecoverable; (2) the `(full … saved to .baxter/outputs/…)` note was appended then passed through head-only `trunc_chars`, so a large body clipped the note off and killed the retrieval path; (3) the empty/unwritable-root path still did destructive compaction while claiming retrievability. | For any feature whose safety story is "lossy inline view + full stash for retrieval," verify three things before dispatch: (a) **stash the RAW source**, not a display/filtered projection — trace what the projection drops and confirm the stashed bytes contain everything it lost; (b) **the retrieval pointer must survive truncation** — never append a load-bearing pointer to a body that is then head/tail-truncated; reserve budget (`cap.saturating_sub(note.len())`) and append the note AFTER the cap (or prepend it), with a test that a max-size body still ends/starts with the pointer; (c) **gate the destructive transform on stash SUCCESS** — if the backup write fails or is unavailable, fall back to the prior NON-destructive behavior (plain truncation keeping a contiguous prefix), so "transformed ⇒ retrievable" is an invariant, not a hope. Also: when adding the treatment to a success path, check whether the ERROR path needs it too (large error diagnostics are outputs), and gate the error path on ITS OWN lossy condition (its display cap actually dropping content), not a threshold copied from the success path. |
| C5 | **A test that drives a LIVE external dependency (on-disk DB, real index, network, daemon) sits in the DEFAULT test suite behind only an "absent → skip" guard — so it is fast/green when the dependency is ABSENT but runs SLOW/nondeterministically (or hangs) when the dependency is PRESENT, making `cargo test`/CI non-hermetic.** The author reasons "it skips cleanly when absent," but the skip-guard only fires on the ABSENT branch; on the machine where the resource exists (the dev box, a warm CI cache), the test executes the real expensive path inside the default run. Caught on the Axon symbol-precision diff: an in-crate test called the real retriever → `AxonService::open` on the on-disk RocksDB index; it "skipped" only when the index was missing/locked, but on an indexed workspace `cargo test` sat >60s on it — and local runs that hit a held DB-lock skip path masked the hang. Fix: mark it `#[ignore = "requires a real index; run with --ignored"]` / env-gate / feature-gate so it is OPT-IN, never in the default suite — keep the absent-resource skip-guard as belt-and-suspenders for the opt-in run. | For any test that touches a live external resource (real DB/index/filesystem-state/network/daemon — not an in-memory or temp fixture), it must NOT run in the default `cargo test`/`pytest`/CI suite. An "if absent, skip" guard is insufficient: it only covers the absent branch, so the test is non-hermetic exactly where the resource exists. Gate it behind `#[ignore]` / a feature flag / an env var so the default suite is deterministic regardless of local state, and keep the skip-guard for the explicit opt-in invocation. Ask: "if this resource IS present and slow, does the default suite still finish fast and deterministically?" If not, it's not a default-suite test. (This is RUNTIME hermeticity of a test that compiles fine — beware the inverse-masking trap: a locked/absent local resource makes it skip, hiding the hang from the author.) |
| C10 | **A security guard that ENUMERATES a command's option grammar to resolve a command-position (allow/deny) models only a SUBSET of each flag's spellings — short flag but not its long alias, or it omits a NON-EXEC mode — so a catastrophe behind the unmodeled spelling dilutes past the guard, and a non-exec mode mis-parses a benign trailing token into a false catastrophic argv0 (over-deny).** When a classifier hand-models a CLI's options to decide whether the rest is a wrapped command (exec-wrapper unwrap, sudo/privilege detection, an interpreter `-c` carve-out), modeling `-x` but not `--exclusive`, `-c` but not `--class`, `-w` but not `--wait` leaves the long form unhandled → the wrapped catastrophe falls to the ambiguous branch (here ASK, but in an allow-context could be silent). Symmetrically, treating a NON-EXEC selector (`chrt -p PID`, `ionice -p`, `chrt -m`/`--max` print-and-exit, `flock -c STRING`) as if it wrapped a command mis-binds a positional/trailing token as the inner argv0 → over-deny. Caught on the Baxter EXEC_WRAPPER unwrap: the first impl modeled short flags only; Codex found `flock --exclusive … rm -rf /`, `env --ignore-environment sudo reboot`, `setsid --wait`, `chrt --fifo`, `ionice --class`, `xargs -J` all diluting deny→ask, plus `chrt -m`/`ionice -p` over-denying. | For any guard that hand-models a command's option grammar, enumerate EACH modeled flag's COMPLETE spelling set — short `-x`, long `--alias`, `--long=value`, and combined `-xVALUE` — from the actual man page (don't model one spelling and assume the rest); a missed alias is a silent grammar hole. Separately, list the command's NON-EXEC / print-and-exit / operate-on-existing-PID / packed-string modes and route them to the SAFE-ambiguous state (ASK/None), never unwrap them as if a command follows. Verify the dominance property both ways: no catastrophe slips via an unmodeled common spelling (dilution), and no benign non-exec form newly denies (over-deny). Pairs with P33 (ambiguous→ask, not deny-on-guess) — C10 is the enumeration-completeness half: P33 says "don't guess," C10 says "model the whole real grammar so you don't have to." |
| C11 | **A plan that adds an ALTERNATIVE score/weight/priority (`rerank_score`, an override, a boost) and threads it through EXISTING selection code via `new.unwrap_or(old)` has three silent failure modes the plan usually misses: (a) it must reach EVERY raw-old-value read site or the feature is PARTIALLY INERT; (b) refactoring an existing tie-break (e.g. first-wins "keep first chunk == best within EPSILON") into `max_by(new)` changes the chosen element ON TIES even when the new value is absent, breaking any "byte-identical when off" claim; (c) a non-finite `new` value (NaN/Inf from a model/division) POISONS the `>` and `partial_cmp(..).unwrap_or(Equal)` comparators it now flows into — `NaN > x` is false and `(NaN-best).abs()<EPS` is false, so a selector can keep NOTHING.** Caught on the Axon reranker spike design: file survival is decided by raw `chunk.score` in THREE places (per-file dedup best-chunk `budget.rs:194-215`, max-files eviction `:228-240`, budget greedy `:108-130`); the plan named only "dedup + budget" (two), and the "net-zero when off" story rested on a refactor that, written naively as `max_by`, would have changed dedup's representative chunk on score ties with the reranker disabled. NaN from the new f32 source would silently empty a file's chunk set. | For any plan adding an alternative/override value consumed by existing ranking/selection/eviction code: (1) grep EVERY read of the old field across the selector(s) and confirm the effective-value reaches ALL of them — list the sites by file:line, don't say "the selection logic" (a missed site = partially-inert feature, the inverse of a no-op); (2) centralize as ONE accessor (`effective_x()`) used by all comparators, never hand-threaded `unwrap_or` per closure; (3) preserve the EXISTING tie-break semantics verbatim — if the code is "first-wins within EPSILON," keep first-wins (swap only the value read), do NOT swap to `max_by`/`sort` which reorders ties; (4) guarantee the new value is FINITE at the write boundary (clamp/reject NaN/Inf) before it enters any `>`/`partial_cmp` site, at a SINGLE sanitization boundary — do NOT split it across producer+caller with contradictory contracts (a producer that remaps `NaN→NEG_INFINITY` is still emitting a non-finite value whose doc claims "finite only"; pick one site, usually the caller, and have it reject non-finite → fall back to the old value); (5) the "byte-identical when off" claim needs a dedicated pre-feature-vs-feature-disabled equality test — an upstream golden (here `fuse()`) or a sibling-parity test does NOT cover the post-feature selector refactor, AND a run-to-run determinism / self-comparison test is ALSO insufficient (it compares post-change code to itself and misses regressions the new wiring introduced) — you must capture a FROZEN pre-edit baseline snapshot BEFORE any selector edit lands and assert exact equality against that constant. Pairs with P30 (read the helper BODY) and P31 (None-sentinel semantics): C11 is the write side — a new optional value flowing INTO existing comparators. |
| C12 | **A cross-process lock built on a WALL-CLOCK TTL guarding a SYNCHRONOUS (no-await) critical section is a nest of distinct concurrency bugs that surface one at a time — unconditional release stomps a reclaimer's lock, a `setInterval` "heartbeat" never fires (the event loop is blocked by the synchronous holder), a fixed TTL evicts a still-LIVE long-running owner, and the read→delete→insert reclaim is a cross-process TOCTOU.** Each is real and distinct; fixing one reveals the next. Caught on engram ADR-001 sync run-lock across many Codex rounds: (1) `release()` did `DELETE WHERE id=1` unconditionally → after a TTL-reclaim it deleted the new owner's row → needed an owner LEASE TOKEN, `release` scoped `WHERE token=?`; (2) the heartbeat was `setInterval(renew, …)` but `sync()` has zero `await` so the timer can't run → had to renew INLINE in the work loop; (3) inline renew still didn't cover non-loop phases, and ANY fixed TTL reclaims a live owner whose sync legitimately exceeds it; (4) the correct model is LIVENESS-based — store `owner_pid`, reclaim only when `!isProcessAlive(pid)` (`process.kill(pid,0)`/ESRCH), never on age; (5) even then the dead-row reclaim `DELETE WHERE id=1` races cross-process (two callers read the same dead pid, A reclaims, B's delete wipes A's fresh row) → reclaim must be COMPARE-AND-SWAP `DELETE WHERE id=1 AND owner_pid=? AND locked_at=?` on the OBSERVED values. | For any cross-process mutex in SQLite (or any check-then-act lock): (a) NEVER release/reclaim with an unconditional `DELETE WHERE <pk>` — scope every delete to the identity you hold/observed (lease token for release; observed `owner_pid`+`locked_at` for reclaim) so it's a compare-and-swap, not a blind wipe; (b) do NOT use `setInterval` to keep a lock alive across a SYNCHRONOUS critical section — the event loop is blocked, the timer is dead code (refresh inline at work checkpoints, or don't rely on refresh); (c) prefer LIVENESS (owner PID + `kill(pid,0)`) over a wall-clock TTL — a fixed TTL either evicts a live long-running owner (overlap) or strands a crash (pick your failure), whereas liveness never evicts a live owner and reclaims a dead one immediately; keep a generous TTL only as a documented PID-reuse backstop, and even then gate it so it can't override a confirmed-live owner; (d) make table creation `CREATE TABLE IF NOT EXISTS` so two first-runs on an upgraded DB don't race on the check-then-create. Note: in a single process better-sqlite3 is synchronous so same-process callers serialize — the races are all CROSS-process (multiple daemons/CLIs on one DB file), which is exactly when the lock matters. |
| C13 | **A change that flips a DEFAULT MODE (or adds a behavior gate) gates the MAIN code path but leaves SIDE-EFFECT branches that also act under the old default ungated — so the headline invariant the flip advertises is silently violated by a less-traveled branch.** The author flips the obvious loop and writes a test for the common path; the rare side-branches (deletions, cleanup passes, dry-run preview counters, error paths) still run the old behavior. Caught on engram ADR-001 pull-by-default: the flip + the marquee "pull makes ZERO vault writes" (AC-2) test covered create/update, but `deleted-on-engram` still called `moveToTrash` (engram→vault) on pull, the `nosync`-cleanup pass still trashed files on every non-dry sync, AND the dry-run `deleted.engram++` still counted a deletion the real pull wouldn't do — three separate AC-2 leaks the same-model audit missed because the test only exercised the main path. | For any default-flip / new behavior-gate, grep EVERY site that performs the gated side effect (here: every vault-mutating call — `moveToTrash`/`writeFile`/`unlink`/`rename`), not just the primary loop, and confirm each is under the SAME gate (or is genuinely in the other direction / DB-only). Enumerate the rare branches explicitly: deletions, tombstones, cleanup/GC passes, conflict paths, AND the dry-run/preview counters (a preview that reports an action the real run now skips is itself a bug). Write the invariant test to drive each side-branch (delete-on-X, tag-gains-flag, dry-run), not only create/update — the headline invariant ("pull = zero writes", "read-only mode", "feature-flag off") is only as true as its least-traveled branch. |

## Notes for learning capture

- Append a new row only when a finding is **recurring** (would apply to other plans/code), not a
  one-off project-specific logic error.
- If pre-flight missed something Codex then caught, the pattern description here is too weak —
  tighten the existing row rather than adding a near-duplicate.
- Keep "Check before dispatch" mechanical and verifiable — a step someone can actually run, not a
  principle to keep in mind.
