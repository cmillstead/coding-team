# C23

`@tags: test-fixture-completeness; reasoning-shape; scope:both` **A "pins every field" behavioral test asserts each known field of an object/payload INDIVIDUALLY (by value) but never asserts the EXACT top-level KEY SET — so REMOVING or RENAMING an asserted field fails, but ADDING a new (valid) field slips through silently.** The test author proves "these fields have these values" and calls it proven-teeth, but the assertion set is open at the top: an extra key the code starts emitting (or a value leaked from a sibling branch) is invisible because no assertion counts the keys. This is especially sharp when the return type is a `Partial<…>`/open interface (excess-property checking does NOT apply to a spread or a non-fresh variable, so tsc won't catch the addition either) — the type-gate and the per-field test BOTH miss additions. Caught on Mabel 05-01 (Codex challenge F1/F2): the enabled-echo config test asserted all 15 knobs by value + `toEqual` on the nested array/object, but never `Object.keys(agentBody)` — so `ambient_sound: "coffee-shop"` (a valid AgentCreateParams field) passed both tsc and the test; and the disabled-branch omission list enumerated 7 absent keys but missed 4 (`interruption_sensitivity`/`responsiveness`/`enable_backchannel`/`enable_dynamic_responsiveness`), so a leaked `responsiveness` from the disabled return passed. Fix: `expect(Object.keys(obj).sort()).toEqual([...exact set...].sort())` on BOTH branches — closes add + leak in one assertion.

**Check before dispatch:** for any test that claims to PIN a whole config/payload/object ("asserts every field", "proven-teeth", "all audited knobs") by asserting fields one-by-one: does it ALSO assert the exact top-level key SET (a `Object.keys(...).sort()` toEqual, a full-object `toEqual`/`toStrictEqual`, or an explicit absence list that is provably COMPLETE)? If it only asserts the fields it names, ADDING a new key or LEAKING a key from a sibling branch passes silently — the teeth are one-directional. Flag it and add the key-set pin. Two amplifiers to check: (1) if the value's type is `Partial<T>`/an open interface AND it reaches the sink via a spread or a variable (not a fresh object literal at the return), tsc's excess-property check does NOT fire, so the type is NOT a backstop for additions; (2) an absence/omission list (`expect("x" in obj).toBe(false)`) is only as strong as its completeness — enumerate the sibling branch's full key set and diff, don't hand-pick a few.

**Design default:** when a test's job is to pin a whole object, assert the exact key set (`Object.keys(x).sort()` toEqual, or `toStrictEqual` on the whole object), not just the fields you remembered to name — a per-field-only pin catches removals and renames but never additions or cross-branch leaks.
