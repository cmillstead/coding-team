| P34 | `@tags: plan-symbol; reasoning-shape; scope:plan` **Plan labels a `#[cfg(test)]` call site as a PRODUCTION path** — the plan says "thread the config through both production call sites" / "the production callers (`a.rs:N`, `b.rs:M`)" / "second production path", but one of those call sites lives inside a `#[cfg(test)] mod tests` block (or behind a `#[cfg(test)]`/`#[test]` gate) and is exercised only by tests. The real production entrypoint reaches the shared function through a different, single path (e.g. a service method), so the "second" wiring is wasted/confusing work AND the architecture claim is wrong. The author reasons from "function `foo` is called at lines N and M" without checking that line M is under a test module. Caught on the Axon test-aware-anchors plan: `detect_anchors`/`fuse` were called from `service/mod.rs::run_pipeline` (production) AND `axon-cli/.../query.rs:374/431` — but the CLI calls sat inside `#[cfg(test)] mod tests` (module opened at query.rs:216); the CLI's real entrypoint just called `AxonService::query`, which routed through `run_pipeline`. The plan's "wire both production call sites" was a misread; only the service path needed wiring, and the test-gated call sites just needed a default arg to compile. | For every call site a plan calls "production" / "a production path" / "the production caller", open the file and confirm the call is NOT inside a `#[cfg(test)] mod tests` block (Rust), a `#[test]`/`#[cfg(test)]`-gated fn, an `if __name__ == "__main__"` / `tests/`-dir / pytest module (Python), or a `describe`/`it` test block (JS). Trace the actual production entrypoint (CLI `main`/command fn, server handler, public API) and verify it reaches the shared function — often it routes through ONE service method, not the call sites the plan enumerated. A call site under a test gate is test scaffolding: it may need a signature-compat update to compile, but it is NOT production wiring. |

**Design default:** Before calling any call site "production" in a plan, open the file and confirm it is not under a `#[cfg(test)]`/test-module/`tests/` gate — trace the real production entrypoint to the shared function rather than listing every textual call site.
