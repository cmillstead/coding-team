# C20

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C20 | `@tags: test-fixture-completeness; reasoning-shape; scope:both` **A behavioral test asserts an entity is PRESENT in the output, but the fixture seeds NO route for that entity to reach the output** — so the assertion is vacuous: it either always fails, or (worse) the implementer quietly weakens it to make it pass, and the guarantee the test was meant to prove goes untested. The author seeds the entities the test names but forgets the EDGE/relation/signal/input that the asserted code path actually consumes to surface them. Caught on the Axon test-aware-anchors plan: a "demotion, not exclusion" integration test seeded a source symbol and a test symbol but NO graph edge between them and used a no-embedder service — after the anchor filter dropped the test anchor, the test symbol had zero route into candidates (no structural edge → BFS finds nothing; no embedder → no semantic; no commits → no temporal), so `assert!(test_chunk.is_some())` could only fail. The fix was to seed a real source→test edge so the symbol enters as a distance-1 structural candidate that the role-cap demotes — making the demotion actually observable. | For any test (in a plan spec OR a diff) that asserts an entity is PRESENT / surfaced / ranked / returned, trace the data path: what input must the fixture provide for that entity to reach the asserted output? Seed-the-named-symbols is not enough when the asserted behavior depends on a graph edge, a foreign-key relation, an embedding, a config flag, an event subscription, or a prior write. If the fixture provides no such route, the test is vacuous — flag it and add the missing seed (or assert on the reachable signal instead). Ask: "with exactly this fixture, by what concrete mechanism does the asserted entity appear?" If you cannot name the mechanism, the test cannot pass honestly. Beware the silent-weakening trap: a vacuous presence-assert often gets softened to a tautology during implementation. |

**Design default:** For every "entity is present in output" test assertion, name the concrete mechanism (edge/relation/signal/input) by which the fixture causes that entity to surface, and seed it — never rely on seeding the named symbols alone.
