# C13

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C13 | `@tags: default-flip; reasoning-shape; scope:both` **A change that flips a DEFAULT MODE (or adds a behavior gate) gates the MAIN code path but leaves SIDE-EFFECT branches that also act under the old default ungated — so the headline invariant the flip advertises is silently violated by a less-traveled branch.** The author flips the obvious loop and writes a test for the common path; the rare side-branches (deletions, cleanup passes, dry-run preview counters, error paths) still run the old behavior. Caught on engram ADR-001 pull-by-default: the flip + the marquee "pull makes ZERO vault writes" (AC-2) test covered create/update, but `deleted-on-engram` still called `moveToTrash` (engram→vault) on pull, the `nosync`-cleanup pass still trashed files on every non-dry sync, AND the dry-run `deleted.engram++` still counted a deletion the real pull wouldn't do — three separate AC-2 leaks the same-model audit missed because the test only exercised the main path. | For any default-flip / new behavior-gate, grep EVERY site that performs the gated side effect (here: every vault-mutating call — `moveToTrash`/`writeFile`/`unlink`/`rename`), not just the primary loop, and confirm each is under the SAME gate (or is genuinely in the other direction / DB-only). Enumerate the rare branches explicitly: deletions, tombstones, cleanup/GC passes, conflict paths, AND the dry-run/preview counters (a preview that reports an action the real run now skips is itself a bug). Write the invariant test to drive each side-branch (delete-on-X, tag-gains-flag, dry-run), not only create/update — the headline invariant ("pull = zero writes", "read-only mode", "feature-flag off") is only as true as its least-traveled branch. |

**Design default:** When flipping a default or adding a behavior gate, gate every site with the side effect — deletions, cleanup/GC passes, and dry-run counters — not just the primary loop, before claiming the invariant holds.
