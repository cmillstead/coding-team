# P33

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| P33 | `@tags: ambiguity-guard; reasoning-shape; scope:both` **A security/validation guard that resolves AMBIGUITY by enumerating candidate interpretations and DENYING if ANY candidate is dangerous over-denies inputs whose BENIGN interpretation was equally valid.** "A false-positive deny is free because the dangerous reading is malicious" is FALSE whenever a benign reading produces the same token shape — the guard then hard-blocks legitimate commands, eroding trust in the gate (the exact "the gate is dumb, stop trusting it" failure the gate exists to prevent). Caught on the Baxter exec-wrapper plan: a ±1 candidate model for unmodeled wrapper flags emitted two candidate command-positions and denied if either was a deny-family command; `timeout --weird echo sudo` (benign — `echo` prints the word "sudo") denied because `sudo` landed in a candidate slot, even though the real command was `echo`. The premise "the string is malicious" held for `timeout --weird 5 sudo reboot` but not for the token-identical benign case. | For any guard that DISAMBIGUATES by guessing among candidate parses/positions/interpretations, do NOT "deny if any candidate is dangerous" — that over-denies every input whose benign parse shares the shape. Deny ONLY on POSITIVE detection: an UNAMBIGUOUS resolution to a dangerous value. When the parse is ambiguous, fall to the safe NON-deny state — for a 3-way classifier that is ASK (human-gated), never DENY-on-a-guess and never silent ALLOW. Test the dominance property: the new rule must introduce ZERO over-deny relative to the prior behavior while only tightening positively-detected cases. The tell is a rationalization like "a false-positive deny is free / the string is malicious anyway" — verify a BENIGN input can't produce the same candidate shape before accepting it. |
