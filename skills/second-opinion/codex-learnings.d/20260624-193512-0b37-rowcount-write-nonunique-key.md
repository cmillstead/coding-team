# C21

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C21 | `@tags: uniqueness-enforcement; tenant-isolation; reasoning-shape; scope:both` **A row-locating WRITE/stamp (`UPDATE … WHERE key = $k`, or "find by key then assert single") relies on single-row semantics — `rowCount === 1`, a `{persisted}`/`{claimed}` boolean, "lookup returns ≤1" — on a key column that is NOT backed by a UNIQUE constraint, so a duplicate key value silently affects MULTIPLE rows and the boolean result LIES (returns `false` on a multi-row write while having actually written N).** The author reasons "this id is unique in practice" (a provider mints globally-unique ids) and adds only a NON-unique lookup index, then asserts `rowCount === 1`. A duplicate — from corruption, a replay, a reconciliation insert — makes the write non-deterministic in row count: a 2-row stamp returns `{persisted:false}` (an apparent MISS) having written two rows, or a "claim" returns true having over-written. On a MULTI-TENANT table the enforcing index must be UNIQUE on `(tenant_col, key)`, not the bare key — a global unique key on an external id is correct-by-coincidence and silently drops a colliding tenant's row. Caught on Mabel M1 03-01: `recordEmergencyEvent` (a life-safety audit stamp) located the `call` row by `(location_id, retell_call_id)` with only a non-unique lookup index; a duplicated `retell_call_id` within a location would stamp >1 row → `rowCount ≠ 1` → `{persisted:false}` reported despite writing. Fix: `CREATE UNIQUE INDEX … ON call (location_id, retell_call_id) WHERE retell_call_id IS NOT NULL`. Same class as the 01-03→0006 tenant-scoped-idempotency-key fix. | For any write/stamp/claim that LOCATES rows by a key and relies on single-row semantics (`rowCount === 1`/`=== 0`, a `persisted`/`claimed`/`updated` boolean derived from rowcount, or a "find-by-X returns at most one" assumption), open the SCHEMA and confirm a UNIQUE constraint/index actually ENFORCES that key's uniqueness — an "expected unique" comment, a `RETURNING id` on a presumed-single row, or a "the provider mints globally-unique ids" rationale is NOT enforcement. If the table is multi-tenant, the UNIQUE key MUST include the tenant column (`UNIQUE (tenant_col, key)`); a bare-key unique constraint either leaks across tenants or rejects a legitimate same-key row in another tenant. For a PARTIAL index, the conflict/predicate must be restated. If the locating key is not uniquely enforced, the single-row assertion is unsound: make it UNIQUE, or handle `rowCount > 1` explicitly (never let a boolean silently misreport). Pairs with C16 (tenant-isolation — there the gap is a missing scope PREDICATE on reads; here it's missing uniqueness ENFORCEMENT on the scoped write key) and the tenant-scoped-idempotency-key memory. |

**Design default:** When a row-locating write asserts single-row semantics (`rowCount === 1`, a `persisted`/`claimed` boolean), enforce the locating key with a UNIQUE constraint — on a multi-tenant table, `UNIQUE (tenant_col, key)` (partial-predicate restated for a partial index) — never rely on an "expected unique" comment or a globally-unique-id rationale.
