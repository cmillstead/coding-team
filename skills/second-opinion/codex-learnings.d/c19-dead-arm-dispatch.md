# C19

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C19 | `@tags: default-flip; command-grammar; reasoning-shape; scope:both` **A plan/diff that ADDS a new match-arm / guard / deny-case assumes its inputs REACH it, but an EARLIER branch in the same dispatch function `return`s first for exactly those inputs — so the new arm is dead for the cases the plan most cares about, and a corpus row asserting the new behavior is wrong about current AND post-change reality.** The author reasons about the new arm in isolation ("shells piped from a network producer deny at the network arm") and never traces whether control flow actually arrives there; an interceptor/early-dispatch branch (a wrapper-unwrap, a shell-string recursion, a special-case `if let Some(..) = .. { return .. }`, a `match` arm ordered above the new one) consumes the input and returns before the new arm runs. Caught on the Baxter G4 plan (Codex round 1): the plan/design asserted `curl | bash -c 'echo hi'` was an "unconditional shell deny" and listed it as a no-regress DENY row, but at `shell_classify.rs:1190-1198` a shell WITH `-c` is intercepted by the 06-03 shell-string-recursion branch and `return`s BEFORE the network arm at `:1236` — so the real classification is recurse-on-inner (`echo hi` → benign → ASK), the opposite of the plan's claim. Two same-model reviews (a plan-doc reviewer that verified every other source line, and the author's own analysis) both missed it because they confirmed the new arm's LOGIC without tracing its REACHABILITY. | For any plan/diff that adds a guard/deny-case/match-arm to an existing dispatch function, TRACE the function top-to-bottom for the inputs the new arm targets and confirm NO earlier branch `return`s first: list every `return`/early-exit ABOVE the new arm (wrapper unwraps, recursion interceptors, `if let Some(..) { return }`, higher `match` arms, guard clauses) and check whether each fires for the new arm's target inputs. If an earlier branch preempts them, the new arm is dead — either move it earlier, fold the logic into the preempting branch, or correct the plan's claim (and any corpus/assertion row) to the ACTUAL post-dispatch behavior. A "new-deny" / "no-regress" row must be validated by running the classifier (or hand-tracing the full dispatch order), not by reading the target arm alone. Pairs with C13 (there an existing side-branch keeps the OLD behavior; here a new arm never RUNS) and P30 (read the body, not the signature — here: read the control-flow ORDER, not just the arm). |
