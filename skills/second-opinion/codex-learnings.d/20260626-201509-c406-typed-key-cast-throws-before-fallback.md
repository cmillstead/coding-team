# C22

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C22 | `@tags: input-shape-guard; reasoning-shape; scope:both` **A resolver/guard whose PRIMARY lookup keys a STRONGLY-TYPED column (`uuid`/`int`/`enum`/`timestamptz`) on UNVALIDATED external text THROWS at the DB driver (Postgres `22P02 invalid input syntax`, or a parse/cast error) on a malformed value — BEFORE the intended FALLBACK path runs — converting a RECOVERABLE case into `internal-error`/5xx and masking the fallback entirely.** The author reasons "primary id present → look it up, else fall back," but a malformed primary value never reaches the `else`: the typed-column comparison (`WHERE id = $text`, `eq(table.uuidCol, val)`, `::uuid` cast) raises inside the query, the surrounding try/catch maps it to internal-error, and the perfectly-valid fallback key (a second field that WOULD have resolved) is never tried. The tell is a two-source resolver — "use A if present, else resolve via B" — where A is a typed column and the A-value is attacker-/vendor-influenceable free text (a webhook metadata echo, a query param, a header). Caught on Mabel M1 04-03 (Codex R2): `resolvePhoneNumberId(ctx, {metadataId, toNumber})` queried `SELECT … WHERE id = ${metadataId}` on the `uuid` PK; a non-UUID `metadata.phone_number_id` (the Retell-echoed, forgeable metadata) would throw 22P02 before the `e164 = toNumber` fallback, turning a recoverable malformed-metadata + valid-number payload into internal-error. | For any resolver/guard with a PRIMARY typed-column lookup + a FALLBACK, confirm the primary value is SHAPE-VALIDATED before the typed query: a `uuid` needs a UUID-regex/`isUuid` guard, an `int` needs an integer parse, an `enum` needs membership — when the guard FAILS, skip straight to the fallback, never let the malformed value reach the typed comparison. Trace the failure path: "if the primary value is garbage, does control reach the fallback, or does the DB driver throw first?" If the column is strongly typed and the input is external free text, it throws first unless guarded. Add a test with a MALFORMED primary value + a valid fallback value asserting the fallback resolves (no throw, no internal-error). Binding the value as a parameter does NOT save you — `text = uuid` still casts and throws; the guard must be in app code BEFORE the query. Pairs with P33/C10 (fail toward the safe/recoverable state, not a hard error) and the verify-before-trust baseline (an external id is untrusted shape AND untrusted ownership).

**Design default:** Any resolver with a primary typed-column lookup (uuid/int/enum) on external text must shape-guard the value in app code before the query and fall through to the secondary key on a guard failure — never let a malformed primary value reach the typed comparison, where it throws and masks the fallback.
