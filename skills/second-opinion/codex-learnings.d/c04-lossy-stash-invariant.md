# C4

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C4 | `@tags: lossy-stash; reasoning-shape; scope:both` **A "lossy transform made safe by a full-fidelity stash/backup" feature breaks its own retrievability guarantee in three coupled ways — stashing an already-LOSSY projection instead of the raw source, appending the retrieval POINTER to a body that is then HEAD-truncated (clipping the pointer when output is largest), and running the destructive transform even when the stash write SILENTLY FAILED.** When a feature compresses/redacts/summarizes data and claims "the full version is retrievable via the stash," all three invariants must hold or the claim is false exactly in the worst case. Caught on the Baxter MCP-output `compact_json` plan: (1) it stashed an already-lossy text projection (non-text content → `"[type content omitted]"`, `structuredContent` dropped) instead of `serde_json::to_string(&res)`, so many results were unrecoverable; (2) the `(full … saved to .baxter/outputs/…)` note was appended then passed through head-only `trunc_chars`, so a large body clipped the note off and killed the retrieval path; (3) the empty/unwritable-root path still did destructive compaction while claiming retrievability. | For any feature whose safety story is "lossy inline view + full stash for retrieval," verify three things before dispatch: (a) **stash the RAW source**, not a display/filtered projection — trace what the projection drops and confirm the stashed bytes contain everything it lost; (b) **the retrieval pointer must survive truncation** — never append a load-bearing pointer to a body that is then head/tail-truncated; reserve budget (`cap.saturating_sub(note.len())`) and append the note AFTER the cap (or prepend it), with a test that a max-size body still ends/starts with the pointer; (c) **gate the destructive transform on stash SUCCESS** — if the backup write fails or is unavailable, fall back to the prior NON-destructive behavior (plain truncation keeping a contiguous prefix), so "transformed ⇒ retrievable" is an invariant, not a hope. Also: when adding the treatment to a success path, check whether the ERROR path needs it too (large error diagnostics are outputs), and gate the error path on ITS OWN lossy condition (its display cap actually dropping content), not a threshold copied from the success path. |
