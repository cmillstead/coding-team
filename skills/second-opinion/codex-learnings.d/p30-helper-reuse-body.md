# P30

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| P30 | `@tags: helper-reuse; reasoning-shape; scope:both` **A plan that REUSES/extends an existing shared helper assumes it WORKS as its name/signature implies, but the helper's BODY is a no-op or contract-violating ā€” an `acquire()` that always returns `true`, a `lockName`/`scope` param accepted-but-ignored, a `validate()` that never rejects, a backing table whose `CHECK`/`UNIQUE`/`PK` forbids the new use.** The symbol EXISTS (P1 passes) and the signature FITS (P2 passes), so name+shape checks wave it through ā€” but building on it inherits a silent defect. Caught across 2 rounds on the engram ADR-001 sync-lock plan: the plan said "reuse `SqliteLockProvider`, key on `lockName`, acquire 'sync'", but `SqliteLockProvider.acquire(lockName, ttlMs)` IGNORES `lockName` and hardcodes `INSERT OR IGNORE INTO consolidation_lock (id) VALUES (1)` then returns `true` UNCONDITIONALLY (never `false` on contention ā€” broken as a mutex), with an existing test pinning that broken behavior; AND the target `consolidation_lock` had `id INTEGER PRIMARY KEY CHECK (id = 1)` in a DEPLOYED (immutable) migration, so per-name keying was structurally impossible without a new migration. The "reuse" would have shipped a lock that never locks. | For any plan that REUSES or EXTENDS an existing helper/utility/provider/table, open its BODY (not just its signature): (1) is every parameter actually USED inside the fn? (grep the param name in the body ā€” an accepted-but-ignored arg is the tell); (2) does the return value VARY with outcome? (a mutex `acquire` must be able to return false; a `find` that always returns the first row; a guard that always passes); (3) does the backing DDL PERMIT the new use? ā€” open the `CREATE TABLE`/constraints; a `CHECK`/`UNIQUE`/`PK (id=1)` may forbid it, and if the DDL lives in a DEPLOYED migration it is immutable (needs a NEW migration, never an edit); (4) does an existing TEST pin the current (possibly broken) behavior? ā€” reuse means updating that test. "It already exists with the right signature" is P1/P2-clean but is NOT proof the body honors the contract its name advertises. Prefer a self-contained implementation over reusing a helper whose body you have not read. Pairs with P1 (phantom symbol ā€” there the symbol is ABSENT; here it's PRESENT but lying) and C9 (type-valid-but-degenerate value). |

**Design default:** Before writing a plan that reuses any helper, open its body and verify: every parameter is used, the return value varies with outcome, the backing DDL permits the new use, and no existing test pins broken behavior ā€” "right signature" is not enough.
