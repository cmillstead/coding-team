# C16

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C16 | `@tags: tenant-isolation; reasoning-shape; scope:both` **Deferring tenant/graph-isolation on a table by arguing "FK `ON DELETE CASCADE` bounds its rows to the parent's tenant" conflates row LIFETIME with row ACCESS — the cascade only deletes child rows when the parent dies; it does NOTHING to scope SELECT/UPDATE/DELETE-by-global-id, which remain cross-tenant.** A table whose every read returns all tenants' rows, or whose mutation addresses a row by a global autoincrement id, is a cross-tenant read/write hole regardless of any FK. Caught on engram P0 sync graph_id plan: `sync_conflicts` was deferred ("FK-CASCADE bounds it transitively"), but `getConflicts()`/`getStatus()` read ALL graphs' conflicts and `resolveConflict(conflictId, ...)` could mark ANOTHER graph's conflict resolved by global id — a cross-tenant WRITE, the exact class the PR existed to close. The deferral rationale was factually wrong and would have shipped the hole. | When deciding whether a table needs a `graph_id`/tenant column, do NOT accept "an FK already bounds it" — FK CASCADE governs row deletion on parent-delete, not query scope. Ask the access questions instead: does any SELECT return rows across tenants? does any UPDATE/DELETE address a row by a GLOBAL id (autoincrement PK) without a tenant predicate? If yes to either, the table needs the scope column and every query needs the predicate — it is in scope, not deferrable. A table slated for future removal is still a live hole until removed. Enumerate ALL tenant-access paths, not only repo/manager query methods — schema-level triggers (AFTER INSERT, AFTER UPDATE) and scheduled prune/GC/retention jobs that SELECT or DELETE without a tenant predicate are cross-tenant holes a method-by-method audit walks past; e.g. an unscoped `AFTER INSERT` prune trigger (`DELETE ... OFFSET N` with no tenant predicate) is a cross-tenant DELETE invisible to a method enumeration. Pairs with C13 (default-flip leaves a side-branch ungated) — both are "the headline isolation/invariant is only as true as its least-scoped access path." |
