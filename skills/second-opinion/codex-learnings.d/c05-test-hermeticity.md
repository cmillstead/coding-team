# C5

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C5 | `@tags: test-hermeticity; provable; scope:diff` **A test that drives a LIVE external dependency (on-disk DB, real index, network, daemon) sits in the DEFAULT test suite behind only an "absent → skip" guard — so it is fast/green when the dependency is ABSENT but runs SLOW/nondeterministically (or hangs) when the dependency is PRESENT, making `cargo test`/CI non-hermetic.** The author reasons "it skips cleanly when absent," but the skip-guard only fires on the ABSENT branch; on the machine where the resource exists (the dev box, a warm CI cache), the test executes the real expensive path inside the default run. Caught on the Axon symbol-precision diff: an in-crate test called the real retriever → `AxonService::open` on the on-disk RocksDB index; it "skipped" only when the index was missing/locked, but on an indexed workspace `cargo test` sat >60s on it — and local runs that hit a held DB-lock skip path masked the hang. Fix: mark it `#[ignore = "requires a real index; run with --ignored"]` / env-gate / feature-gate so it is OPT-IN, never in the default suite — keep the absent-resource skip-guard as belt-and-suspenders for the opt-in run. | For any test that touches a live external resource (real DB/index/filesystem-state/network/daemon — not an in-memory or temp fixture), it must NOT run in the default `cargo test`/`pytest`/CI suite. An "if absent, skip" guard is insufficient: it only covers the absent branch, so the test is non-hermetic exactly where the resource exists. Gate it behind `#[ignore]` / a feature flag / an env var so the default suite is deterministic regardless of local state, and keep the skip-guard for the explicit opt-in invocation. Ask: "if this resource IS present and slow, does the default suite still finish fast and deterministically?" If not, it's not a default-suite test. (This is RUNTIME hermeticity of a test that compiles fine — beware the inverse-masking trap: a locked/absent local resource makes it skip, hiding the hang from the author.) |

**Design default:** Any test that touches a real external resource (on-disk DB, network, daemon) must be opt-in (`#[ignore]` / env-gate / feature-flag) from the start — never default-suite with an "absent→skip" guard as the sole hermeticity control.

**Partial mechanization (stays live — all-advisory):** A v0.5 spike measured per-language detection (real corpus: 105 Rust files / 984 test-fns, 130 Python files / 2550 test-fns) and proposed Rust as a commit-time block. The post-apply Codex gate then found five successive false-block edges in the Rust line-parser (multi-line attributes, string-literal contents, comment brackets, multi-line-string braces, commented-out `#[test]` markers) — each a real false-positive, which is the one disqualifying axis for a blocker. The spike's FP=0 held only because the real corpus under-sampled Rust's syntax; the false-positive surface is open-ended for a line-parser. **C5 therefore ships ALL-ADVISORY:** both Rust and Python are author-time nudges via `graduated_checks.py` (PreToolUse Edit|Write) — Rust gated by the `.rs` extension, Python by test-file path — never a commit block. An advisory false-positive is a harmless spurious nudge, so the FP arms race is moot and the conservative-gate rule is no longer load-bearing; the detector stays a best-effort line-parser that may over- or under-fire on adversarial syntax, which is acceptable for an advisory. It does **not** guarantee hermeticity. The full thesis — structural/external-vs-hermetic discrimination beyond the syntactic signal — remains owned by **Codex review, by design.** C5 does **not** graduate or relocate; it stays live with this truthful marker.

**Findings (v0.5 blocking pilot):** (1) **Blocking viability needs more than a positive signature plus a real positive corpus — it needs detection that does not depend on hand-parsing the host language.** Rust C5 cleared the spike (positive `#[ignore]`-on-real-open signature, four gold positives) yet still failed blocking, because robustly distinguishing the violation requires correctly parsing Rust — comments, string literals, brace tracking, and macros are exactly what the line-parser kept getting wrong. If Rust C5 blocking is ever revived it must be built on a real AST (`syn`), never a line-parser; that is its own scoped cost/value decision, not something to start under a blown budget. (2) **A blocking spike's FP corpus must include adversarial/synthetic syntax, not just real-corpus samples.** The spike's FP=0 was real-corpus-only; the post-apply gate then found false-positives on syntax the corpus never contained. For future blocking pilots, adversarial syntax goes into the spike BEFORE the FP≈0 claim is made — not in five remediation rounds after.
