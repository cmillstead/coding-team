# C15

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C15 | `@tags: migration-parity; reasoning-shape; scope:both` **A migration/parity test that recreates the PRE-migration schema as a fixture must reproduce that schema's CONSTRAINTS and ALL its objects (indexes, triggers, defaults) faithfully — an infidelity either makes the test setup IMPOSSIBLE (seed rows that violate the old constraint can't be inserted) or makes the parity gate BLIND (a comparison of only `{name,type,notnull}` misses `dflt_value` drift and skips non-table objects entirely).** Two coupled failure modes from the same root. Caught on engram P0 sync graph_id plan: (a) the legacy fixture built `UNIQUE(vault_path)` then the multi-graph test inserted TWO rows with the same `vault_path` — impossible under that very index (the duplicate must be inserted only AFTER the migration rebuilds it as composite `(graph_id, vault_path)`); the fixture also omitted the real `sync_log` indexes + prune trigger, so upgrade coverage was unfaithful. (b) the parity test compared only `{name,type,notnull}` and a single index — it would miss `dflt_value` divergence (the ALTER path MUST carry `DEFAULT 'x'` because SQLite requires a default for `ADD COLUMN NOT NULL`, while a fresh `CREATE TABLE` can omit it → defaults diverge, `notnull` does NOT catch it) and miss the other indexes + the trigger. A bogus "the notnull bit forces default alignment" note made it worse: SQLite reports `notnull=1` for `x TEXT NOT NULL` WITHOUT any default. | For any migration/parity test: (1) build the legacy fixture as a VERBATIM copy of the real pre-migration DDL — every column default, every index, every trigger, every constraint — not a simplified subset; seed data that the OLD constraints permit, and exercise the NEW constraint's relaxation only AFTER the migration runs (insert the now-legal duplicate post-migration). (2) A schema-parity comparison must cover `dflt_value` (not just name/type/notnull) AND every non-table object (indexes, triggers) via normalized `sqlite_master.sql`. (3) Never reason "`notnull` proves the default matches" — `ADD COLUMN NOT NULL DEFAULT x` and `CREATE ... NOT NULL` both report `notnull=1` but differ on `dflt_value`; the default is the drift axis a column-shape check misses. Pairs with C21-class dual-schema parity (multiple DDL sources must converge). |
