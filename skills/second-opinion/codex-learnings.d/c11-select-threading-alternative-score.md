# C11

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C11 | `@tags: select-threading; metric-aggregate; provable; scope:both` **A plan that adds an ALTERNATIVE score/weight/priority (`rerank_score`, an override, a boost) and threads it through EXISTING selection code via `new.unwrap_or(old)` has three silent failure modes the plan usually misses: (a) it must reach EVERY raw-old-value read site or the feature is PARTIALLY INERT; (b) refactoring an existing tie-break (e.g. first-wins "keep first chunk == best within EPSILON") into `max_by(new)` changes the chosen element ON TIES even when the new value is absent, breaking any "byte-identical when off" claim; (c) a non-finite `new` value (NaN/Inf from a model/division) POISONS the `>` and `partial_cmp(..).unwrap_or(Equal)` comparators it now flows into — `NaN > x` is false and `(NaN-best).abs()<EPS` is false, so a selector can keep NOTHING.** Caught on the Axon reranker spike design: file survival is decided by raw `chunk.score` in THREE places (per-file dedup best-chunk `budget.rs:194-215`, max-files eviction `:228-240`, budget greedy `:108-130`); the plan named only "dedup + budget" (two), and the "net-zero when off" story rested on a refactor that, written naively as `max_by`, would have changed dedup's representative chunk on score ties with the reranker disabled. NaN from the new f32 source would silently empty a file's chunk set. | For any plan adding an alternative/override value consumed by existing ranking/selection/eviction code: (1) grep EVERY read of the old field across the selector(s) and confirm the effective-value reaches ALL of them — list the sites by file:line, don't say "the selection logic" (a missed site = partially-inert feature, the inverse of a no-op); (2) centralize as ONE accessor (`effective_x()`) used by all comparators, never hand-threaded `unwrap_or` per closure; (3) preserve the EXISTING tie-break semantics verbatim — if the code is "first-wins within EPSILON," keep first-wins (swap only the value read), do NOT swap to `max_by`/`sort` which reorders ties; (4) guarantee the new value is FINITE at the write boundary (clamp/reject NaN/Inf) before it enters any `>`/`partial_cmp` site, at a SINGLE sanitization boundary — do NOT split it across producer+caller with contradictory contracts (a producer that remaps `NaN→NEG_INFINITY` is still emitting a non-finite value whose doc claims "finite only"; pick one site, usually the caller, and have it reject non-finite → fall back to the old value); (5) the "byte-identical when off" claim needs a dedicated pre-feature-vs-feature-disabled equality test — an upstream golden (here `fuse()`) or a sibling-parity test does NOT cover the post-feature selector refactor, AND a run-to-run determinism / self-comparison test is ALSO insufficient (it compares post-change code to itself and misses regressions the new wiring introduced) — you must capture a FROZEN pre-edit baseline snapshot BEFORE any selector edit lands and assert exact equality against that constant. Pairs with P30 (read the helper BODY) and P31 (None-sentinel semantics): C11 is the write side — a new optional value flowing INTO existing comparators. |

**Design default:** When adding an alternative score to existing selection code, list every raw-value read site by file:line, centralize via one `effective_x()` accessor, preserve existing tie-break semantics, and sanitize the new value to finite-only at a single boundary before it enters any comparator.
