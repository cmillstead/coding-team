# C17

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C17 | `@tags: path-equality; provable; scope:both` **A helper that decides whether two file paths refer to the SAME file via a bare `ends_with`/substring suffix test (no path-SEPARATOR boundary check) OVER-MATCHES segment-unaligned suffixes — `src/foobar.rs` "matches" `bar.rs`, `crates/axon-rank/src/budget.rs` "matches" `rank/src/budget.rs` — and the bug scales with the helper's cross-reference fan-out.** Bidirectional `a.ends_with(b) || b.ends_with(a)` over normalized paths correctly solves absolute-vs-repo-relative matching (a leading `/` is preserved, so the relative side is a suffix of the absolute side), which is why it passes the obvious positive tests — but it admits two false-match classes: (1) a shorter name that is a non-`/`-aligned tail of a longer one (`foobar.rs` ⊃ `bar.rs`), and (2) two different files sharing a trailing path segment-run with a non-aligned boundary. When the helper is used to attribute/classify by cross-referencing one path against EVERY symbol in a result set (not a single membership test), the false-match surface is large and silently mis-buckets — here it would inflate the exact metric the probe exists to measure (a confirmation-bias trap). "An existing sibling helper uses the same `ends_with`" is NOT license to copy it: the existing one may have a tiny fan-out (one expected-list membership test) while the new one fans out across the whole result set. Caught on the Axon anchor-provenance probe plan: `same_file()` cross-ref'd FP paths against all anchors + structural symbols; Codex flagged the suffix collisions the same-model plan review missed. | For any helper that tests path EQUALITY/identity by suffix, require SEGMENT ALIGNMENT: after normalizing, accept exact-eq OR a suffix match where the character before the matched suffix in the longer path is a `/` (e.g. `long.strip_suffix(short).is_some_and(|p| p.is_empty() || p.ends_with('/'))`), bidirectionally if either side may be absolute. Add NEGATIVE tests for unaligned collisions (`foobar.rs`↔`bar.rs`) and different-dir same-basename (`a/b.rs`↔`c/b.rs`), not just the positive absolute-vs-relative case. Audit the fan-out: a suffix matcher that cross-references against MANY candidates is far higher-risk than a single membership test — "a sibling helper already does it this way" does not transfer when the fan-out differs. Pairs with C1 (path trust boundary — there the risk is traversal/bypass; here it's silent over-match in attribution/classification). |

**Design default:** Any path-equality helper must require segment alignment — accept a suffix match only when the preceding character in the longer path is `/` or the paths are exactly equal — and include negative tests for unaligned collisions like `foobar.rs`↔`bar.rs`.

**Partial enforcement:** Partially enforced by write-guard `check_path_safety` (alias "Case study #35") — a Python-only (`.py`-gated), usage-site, substring-collision advisory (a proxy, suppressed when a safe Path API like `Path(`/`.parts`/`.is_relative_to(` co-occurs); it flags string ops on path-ish variables, it does NOT verify segment alignment. The structural segment-alignment thesis and non-Python comparisons remain owned by cross-model (Codex) review — a deliberate division of labor (hooks catch syntactic usage-site signals; the reasoner owns structural theses), not an un-mechanized oversight. C17 therefore stays a LIVE entry.
