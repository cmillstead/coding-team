# C12

| ID | Pattern | Check before dispatch |
|----|---------|----------------------|
| C12 | `@tags: concurrency-lock; provable; scope:both` **A cross-process lock built on a WALL-CLOCK TTL guarding a SYNCHRONOUS (no-await) critical section is a nest of distinct concurrency bugs that surface one at a time — unconditional release stomps a reclaimer's lock, a `setInterval` "heartbeat" never fires (the event loop is blocked by the synchronous holder), a fixed TTL evicts a still-LIVE long-running owner, and the read→delete→insert reclaim is a cross-process TOCTOU.** Each is real and distinct; fixing one reveals the next. Caught on engram ADR-001 sync run-lock across many Codex rounds: (1) `release()` did `DELETE WHERE id=1` unconditionally → after a TTL-reclaim it deleted the new owner's row → needed an owner LEASE TOKEN, `release` scoped `WHERE token=?`; (2) the heartbeat was `setInterval(renew, …)` but `sync()` has zero `await` so the timer can't run → had to renew INLINE in the work loop; (3) inline renew still didn't cover non-loop phases, and ANY fixed TTL reclaims a live owner whose sync legitimately exceeds it; (4) the correct model is LIVENESS-based — store `owner_pid`, reclaim only when `!isProcessAlive(pid)` (`process.kill(pid,0)`/ESRCH), never on age; (5) even then the dead-row reclaim `DELETE WHERE id=1` races cross-process (two callers read the same dead pid, A reclaims, B's delete wipes A's fresh row) → reclaim must be COMPARE-AND-SWAP `DELETE WHERE id=1 AND owner_pid=? AND locked_at=?` on the OBSERVED values. | For any cross-process mutex in SQLite (or any check-then-act lock): (a) NEVER release/reclaim with an unconditional `DELETE WHERE <pk>` — scope every delete to the identity you hold/observed (lease token for release; observed `owner_pid`+`locked_at` for reclaim) so it's a compare-and-swap, not a blind wipe; (b) do NOT use `setInterval` to keep a lock alive across a SYNCHRONOUS critical section — the event loop is blocked, the timer is dead code (refresh inline at work checkpoints, or don't rely on refresh); (c) prefer LIVENESS (owner PID + `kill(pid,0)`) over a wall-clock TTL — a fixed TTL either evicts a live long-running owner (overlap) or strands a crash (pick your failure), whereas liveness never evicts a live owner and reclaims a dead one immediately; keep a generous TTL only as a documented PID-reuse backstop, and even then gate it so it can't override a confirmed-live owner; (d) make table creation `CREATE TABLE IF NOT EXISTS` so two first-runs on an upgraded DB don't race on the check-then-create. Note: in a single process better-sqlite3 is synchronous so same-process callers serialize — the races are all CROSS-process (multiple daemons/CLIs on one DB file), which is exactly when the lock matters. |
